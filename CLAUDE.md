# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A static **study site** for principal-engineer **system design interview** prep, built with **Astro + Starlight**. Content was migrated from two source PDFs (`principal-engineer-interview-pack.pdf`, `principal_engineer_interview_schoolbook_v2.pdf`, kept in the repo root for reference). It deploys to **GitHub Pages** at `https://jackdo68.github.io/system-design-interview`.

There is no backend and no test suite — it is a content site. The "source of truth" is the Markdown/MDX under `src/content/docs/`.

## Commands

```bash
source ~/.zshrc        # always run before npm in this environment
npm install            # install deps
npm run dev            # dev server — http://localhost:4321/system-design-interview/
npm run build          # production build to dist/ (also builds Pagefind search index)
npm run preview        # serve the built dist/ locally
```

There are no lint or test scripts. `npm run build` is the verification gate — it fails on broken MDX, bad component imports, or invalid frontmatter, so run it after content/config changes.

## Architecture & key conventions

### The base path is `/system-design-interview` — this is load-bearing
`astro.config.mjs` sets `site` + `base: '/system-design-interview'` because GitHub Pages serves this as a *project* page under that sub-path. Consequences:

- **All local URLs are prefixed**: the dev server runs at `http://localhost:4321/system-design-interview/`, not `/`. Hand-typing a path without the prefix 404s locally (GitHub Pages auto-redirects no-slash → slash, but the Astro dev server does not).
- **If the repo is ever renamed**, update `base` in `astro.config.mjs` AND the hardcoded links in `src/content/docs/index.mdx` (see below) AND the GitHub `social` href.

### Internal link rules (the main footgun)
- **Content pages (`.md`)**: link between pages with **relative paths** (`../idempotency/`, `../../deep-dives/sharding/`, always with a trailing slash). Astro rewrites these to include the base. This works because every content page is served as a directory with a trailing slash.
- **`index.mdx` (the splash landing page)**: relative links here are **NOT** base-rewritten (the hero `actions` are frontmatter config; the `<Card>` links are bare relative). So `index.mdx` uses **absolute paths with the base hardcoded**: `/system-design-interview/start/mindset/`. Keep that pattern when editing the landing page.
- Rule of thumb: trailing slashes everywhere; never write a bare `start/mindset` link.

### Sidebar / navigation
The sidebar is **manually defined** in `astro.config.mjs` (`starlight({ sidebar: [...] })`) using `slug` references — it is NOT autogenerated. When you add a page, add its `slug` to the relevant sidebar group, or it won't appear in nav. Section order is controlled both by sidebar array order and by the `sidebar.order` frontmatter on each page.

### Content structure (`src/content/docs/`)
Topic areas map 1:1 to sidebar groups and to folders: `start/`, `concepts/` (the breadth surface), `deep-dives/` (depth), `worked-design/`, `australia-2026/`, `practice/`. The home page is `index.mdx` (`template: splash`).

### Diagrams — Mermaid
`astro-mermaid` is wired in `astro.config.mjs` and **must be listed before `starlight()`** in the integrations array. Write diagrams as fenced ```` ```mermaid ```` code blocks in any `.md`/`.mdx` page; they render client-side with automatic light/dark theming (`autoTheme: true`). No per-page setup needed.

### Callout boxes (asides)
The source PDFs' colored boxes are mapped to Starlight asides via `:::` directives (these work in plain `.md`, no import needed):
- `:::note[Key Idea]` · `:::tip[Principal Move]` · `:::caution[Trap to avoid]` · `:::danger[Never]`

Keep this mapping consistent when adding content — it's the visual identity carried over from the PDFs.

### Theming
`src/styles/theme.css` (registered via `customCss` in `astro.config.mjs`) holds all visual customization: self-hosted Fontsource fonts (Inter body, Lexend display headings, JetBrains Mono code), an indigo/teal accent ramp, card hover-lift, aside and Mermaid framing. Change look-and-feel here, not inline.

## Deployment
`.github/workflows/deploy.yml` builds with `withastro/action@v3` and deploys via `actions/deploy-pages` on push to `main`. Requires repo **Settings → Pages → Source: "GitHub Actions"**. The workflow needs no path config because the Astro project is at the repo root.

## Gotchas
- `npm`-related commands need `source ~/.zshrc` first (environment quirk).
- The `package.json` `name` is still the scaffold default (`sd-site`); harmless, rename if desired.
- The local folder is `system-design-patterns` while the repo/base is `system-design-interview` — cosmetic only, nothing depends on the folder name.
